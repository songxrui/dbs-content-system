# dbs-content-system R3 — 内容结构化工程

> 完整原始文档: `references/original_body.md` (7.8KB)

## 一句话定义

把本地大量内容资产从"堆在文件夹里的库存"变成"可复用、可追溯、可重组、可继续生长的内容结构化工程"。

---

## 5 层工程架构

```
L5: 选题装配稿 (Assembly Drafts)  ← 可发布内容骨架
L4: 主题地图 (Theme Maps)          ← 相关节点组织成知识面
L3: 内容单元库 (Content Units)      ← QST/CON/OPI/CAS/SOL
L2: 状态层 (Status Layer)          ← 已处理/跳过/需复核
L1: 规则层 (Rule Layer)            ← 命名/格式/关联/登记规则
```

---

## 5 类内容单元 (不可再分)

| 类型 | 全称 | 定义 | 示例 |
|------|------|------|------|
| **QST** | Question | 一个问题或痛点 | "为什么大多数人赚不到钱？" |
| **CON** | Concept | 一个概念或定义 | "信息差 = 你知道但别人不知道且能变现的知识" |
| **OPI** | Opinion | 一个有立场的观点 | "先有产品再做内容，不然就是在自嗨" |
| **CAS** | Case | 一个案例或故事 | "李亚鹏认识半个娱乐圈，30年为什么赚不到钱" |
| **SOL** | Solution | 一个可执行的方案 | "每天写3条推文，坚持100天，前30天先别管质量" |

## 4 种关系

| 关系 | 含义 | 示例 |
|------|------|------|
| **回应** | CON 回应 QST | "信息差(CON)解释了为什么赚钱难(QST)" |
| **解释** | CAS 解释 OPI | "李亚鹏案例(CAS)证明了人脉≠赚钱(OPI)" |
| **证明** | SOL 证明 OPI | "100天写作计划(SOL)证明坚持>天赋(OPI)" |
| **冲突** | OPI vs OPI | "先有产品(OPI1) vs 先有流量(OPI2)的冲突" |

## 工程流程 (4 阶段)

### Phase 1: 审计 (先判断，不建工程)
- **数量门禁**: 原始素材 ≥30篇 或 ≥10万字 → 通过
- **来源维度**: 至少包含已发布内容 + 未发布成熟稿 + 外部研究
- **边界清晰**: 用户明确哪些目录纳入/排除
- 不通过 → 输出审计报告，告知用户"当前不适配"，给积累建议

### Phase 2: 骨架搭建 (工程初始化)
1. 创建工程目录 (`01-规则层/` `02-状态层/` `03-内容单元库/` `04-主题地图/` `05-选题装配稿/`)
2. 复制原始素材到 `01-原始素材区/完整副本/` (原始文件不动)
3. 生成 `来源候选.md` `原始索引.md` `待处理清单.md`
4. 建立规则层: 命名规范、字段格式、关联方式、登记规则

### Phase 3: 验证结构 (先抽样本)
- 取 5-10 篇代表性样本进行抽取
- 验证 5 类内容单元边界是否清晰
- 验证 4 种关系判断是否一致
- 结构稳定后 → 进入批量

### Phase 4: 批量推进
- 按优先级: 已发布内容 → 成熟未发布稿件 → 外部研究素材
- 每批 20-50 篇，抽取后立即建立关系链接 (≥50% 覆盖率)
- 每 3 批生成一次主题地图
- 去重候选标注 (相似度阈值默认 0.85)

## 验证清单

## 失败模式与兜底

| 失败模式 | 原因 | 兜底 |
|---------|------|------|
| 内容量不足强行启动 | 用户高估积累 | 输出审计报告，给积累建议后退出 |
| 规则层/状态层未建立就开始抽取 | 跳过 Phase 2 | 回到 Phase 2 建立完整骨架 |
| 内容单元分类混乱 | 边界规则不明确 | 用 5-10 篇样本逐条判定，稳定后再批量 |
| 关系索引空转 | 只抽取不关联 | 每批抽取后立即建立 ≥50% 关系链接 |
